## 1. 评估背景

quorvex_ai 是开源 AI 自愈测试框架，核心思路是 **Browser-based S1/S2/S3 并行修复**   
（来自 2026-07-04 GitHub intelligence scan）。

v5.0 目标：评估 quorvex_ai 的规则引擎思路是否适合硅基体修复策略升级。

## 3. quorvex_ai 与硅基体的关键差异

### 3.1 架构差异

| 维度 | quorvex_ai | 硅基体当前 (v4.0) |
|---|---|---|
| 运行模式 | Browser-based (测试用例) | MES 产线 (HTTP/SDK) |
| 修复目标 | CI/CD 测试失败 | 生产线故障 |
| 输入格式 | 浏览器 DOM / 测试日志 | YAML 结构化故障信号 |
| 回滚机制 | 未知 | Git commit revert (5/5 验证) |
| 安全意识 | 未知 | 熵监控 + 输入消毒 + 路径隔离 |
| 去重 | 未知 | SHA256 + segment hash |

### 3.2 信号粒度差异

**quorvex_ai 信号**: 浏览器错误 / 测试断言失败 / log 字符串（非结构化）

**硅基体信号**: `signals` 字段的结构化键值对（如 `temperature > 980°C`, `burr_height > 20 μm`）

**诊断**: quorvex 的匹配引擎需要 >50% 重写才能适配结构化信号。

### 3.3 并行策略差异

quorvex 的 S1/S2/S3 是 **并行触发**的——同时尝试 3 种修复，取最先成功的。

硅基体当前是 **串行单策略**——根据故障类型固定映射 S1/S2/S3。

**值得借鉴**: 并行策略执行。

## 5. v5.0 可吸收设计点（不 fork，仅借鉴思路）

### 5.1 P0: 并行策略执行

**现状**: `apply_repair()` 固定映射 S1/S2/S3  
**v5.0 改进**: 
```
故障触发 → S1 尝试 + S2 并行准备 + S3 权限预取
          → S1 成功？结束
          → S1 失败？S2 立即启动（已准备）
          → S2 失败？S3  escalation（权限已预取）
```
**价值**: 从 ~3s 单策略延迟降到 ~1s 首策略 + 200ms 故障转移

### 5.2 P1: 回退链

**现状**: 单策略失败 → rollback (全量切换)  
**v5.0 改进**: 策略链回退（S1→S2→S3→escalation）

### 5.3 P2: LLM Fallback (gate 可选)

**现状**: 纯规则引擎，未覆盖的故障无法修复  
**v5.0 建议**: 添加可选 LLM 通道（环境变量 `ENABLE_LLM_FALLBACK=true`），仅当规则引擎返回低置信度时触发

## 6. 结论

| 决策 | 理由 |
|---|---|
| ❌ 不直接 fork/集成 quorvex_ai | 架构不匹配（browser vs MES）、集成成本 3-5 天 |
| ✅ 吸收设计思路 (P0) | 并行策略执行可显著降低修复延迟 |
| ✅ 吸收设计思路 (P1) | 策略链回退提升容错 |
| ⚠️ LLM Fallback (P2) | 保持 gate 可选，避免引入外部依赖 |
| 📌 持续关注 | 如果 quorvex 扩展工业场景，重新评估 |

**v5.0 核心**: 不是引入 quorvex，而是吸收其"并行策略"设计思想，让现有 S1/S2/S3 引擎更高效。
